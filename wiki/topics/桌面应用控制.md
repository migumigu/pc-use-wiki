# 桌面应用控制

> AI Agent控制桌面应用(GUI应用)的技术栈,涵盖Office、IDE、专业软件的自动化操作

## 定义

<!-- confidence: INFERRED -->
桌面应用控制是AI Agent通过鼠标键盘模拟、视觉理解、UI Automation等方式实现对桌面软件(Office、IDE、PS等专业工具)的操作和自动化。

## 技术栈分层

### 协议层
<!-- confidence: EXTRACTED -->
- **UI Automation**: Windows无障碍框架,提供元素定位和操作
- **Accessibility API**: macOS/Linux的无障碍接口
- **Java Access Bridge**: Java应用的无障碍桥接

### 工具实现层
<!-- confidence: EXTRACTED -->
- **PyAutoGUI**: 底层鼠标键盘模拟，跨平台支持
- **AutoHotkey**: Windows专用自动化脚本工具（v2为主版本，持续维护）
- **pywinauto**: Windows UI Automation封装
- **Python-UIAutomation**: Python版UI Automation
- **视觉识别**: OCR、图像匹配

### Agent集成层
<!-- confidence: EXTRACTED -->
- **UI-TARS**: 视觉语言模型驱动的GUI Agent
- **Open Interpreter Computer Use**: 代码执行+桌面控制
- **Anthropic Computer Use**: Claude的桌面控制能力
- **CUA (Computer Use Agent)**: 开源基础设施，提供Drivers+Sandbox+Bench组合方案
- **Bytebot**: 容器化虚拟桌面AI Desktop Agent（Docker+Ubuntu+XFCE）
- **Goose**: Linux Foundation本地AI Agent（MCP+Skills架构）

## 核心技术方法

### 坐标系统方法
<!-- confidence: EXTRACTED -->
- 通过屏幕坐标定位元素
- PyAutoGUI的moveTo、click等API
- 适用场景: 固定布局界面

### UI Automation方法
<!-- confidence: EXTRACTED -->
- 通过元素属性定位(Name、AutomationId)
- Control Patterns标准化操作
- 适用场景: Windows应用、支持Accessibility的应用

### 视觉理解方法
<!-- confidence: EXTRACTED -->
- 截图+OCR识别文字
- VLM理解界面语义
- 适用场景: 所有GUI应用,无需API支持

## 代表性工具对比

| 工具 | 定位方法 | 适用场景 | 优势 | 限制 |
|------|---------|---------|------|------|
| PyAutoGUI | 坐标 | 所有应用 | 简单跨平台 | 易失效 |
| AutoHotkey | 坐标/窗口 | Windows应用 | 脚本强大 | 仅限Windows |
| UI Automation | 元素属性 | Windows应用 | 稳定精准 | 需API支持 |
| UI-TARS | 视觉理解 | 所有GUI | 自然语言 | 需VLM |
| Computer Use | 视觉+模拟 | 所有GUI | AI决策 | 延迟较高 |

## 技术挑战

### 界面变化适应性
<!-- confidence: INFERRED -->
- 坐标方法界面变化后失效
- UI Automation需要应用支持
- 视觉方法实时性挑战

### 精度与稳定性平衡
<!-- confidence: INFERRED -->
- 坐标方法精度高但稳定性差
- 视觉方法稳定性好但精度受限
- UI Automation精度和稳定性都较好

## 部署模式分类

### 本地安装型
直接安装在用户本地桌面上运行：
- [[UI-TARS]]、[[Goose]]、[[Open Interpreter]]
- 优点：深度集成本地环境，响应快
- 缺点：安全性依赖系统，隔离性弱

### 容器化/沙箱型
运行在容器或沙箱环境中：
- [[Bytebot]]（Docker 容器）、[[CUA]]（多沙箱模式）
- 优点：隔离性强，安全可靠，可复现
- 缺点：资源消耗大，本地集成受限
