# 上下文工程

> AI Agent 上下文管理的技术与方法论

## 核心观点

上下文工程将"上下文管理"从被动容量问题转变为主动内容工程。核心目标：解决 AI Agent 上下文窗口污染问题，提升 Token 利用效率。

## 两类问题

### 1. 窗口大小问题

行业主流方向：
```
更大的上下文窗口 —— 128K → 200K → 1M → 2M token
      │
      成本线性增长
```

### 2. 内容质量问题

上下文工程的方向：
```
压缩输入 —— 60-98% token 节省
      │
      在输入到达之前减量
```

## 隐形成本

AI Agent 工作流的典型消耗：

```
一个典型的 SRE 调试会话：
Agent 读取日志文件（17,892 token）
Agent 执行 grep 命令（4,321 token）
Agent 检查配置文件（8,456 token）
Agent 查看指标面板（6,234 token）
Agent 汇总分析（28,791 token）
                           ─────────
总计                      65,694 token
```

30 分钟后，40% 上下文被工具输出占据。

## 四项机制

### 1. Context Saving（上下文节省）

工具输出不进入上下文窗口，只保留引用：

```
原始：315 KB 工具输出
压缩后：5.4 KB
节省：98%
```

### 2. Session Continuity（会话连续性）

所有事件索引到 SQLite + FTS5，对话压缩后检索恢复状态。

### 3. Think in Code

LLM 生成分析脚本而非读取文件：

```
47 × Read() = 700 KB → 1 × ctx_execute() = 3.6 KB
```

### 4. 可逆压缩

原始内容缓存 + 检索恢复：

```python
compressed_prompt = "Log[chunk_id=abc123]: FATAL at 892..."
tool_call("headroom_retrieve", {"chunk_id": "abc123"})
→ 返回原始 10,000 行日志
```
