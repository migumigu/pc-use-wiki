# 浏览器自动化工具对比分析

> 对比 4 个 AI 浏览器自动化工具 | 生成日期：2026-06-28

## 对比对象

- [[agent-browser]] — Vercel Labs，Rust CLI，AI 原生设计
- [[browser-use]] — Gregor Zunic，Python 框架，自然语言驱动
- [[page-agent]] — 阿里巴巴，TypeScript，页面内 GUI Agent
- [[Playwright]] — Microsoft，TypeScript/Python，底层自动化引擎

## 核心差异分析

### 设计哲学对比

1. **agent-browser — AI 原生极简主义**
   - 从命令设计到交互模式都以 AI Agent 为第一用户
   - refs 机制彻底解决选择器不稳定和 Token 浪费问题
   - Rust 原生保证性能，零 Node.js 运行时依赖
   - 牺牲了功能丰富度，换取 AI 友好性和效率

2. **browser-use — 自然语言中间层**
   - 在 Playwright 之上封装自然语言接口
   - 保留 Playwright 的全浏览器支持和丰富功能
   - 引入 DOM 解析和 Prompt 工程降低使用门槛
   - 定位：让 AI 更容易使用传统自动化工具

3. **page-agent — 页面内共生模式**
   - 让 Agent 直接住进网页里，而非从外部操控
   - 零外部依赖，仅需页面内 JavaScript
   - 面向 SaaS 产品开发者，给产品加 AI Copilot
   - 局限性：只能操作当前页面，跨页面需 Chrome 扩展

4. **Playwright — 底层基础设施**
   - 工业级浏览器自动化引擎
   - 全浏览器支持、丰富 API、稳定可靠
   - 面向开发者，需要编写代码
   - 是其他三个工具的底层基础（browser-use 基于 Playwright）

### 技术架构对比

| 架构层级 | agent-browser | browser-use | page-agent | Playwright |
|----------|---------------|-------------|------------|------------|
| 系统层 | Rust + CDP | Python + Playwright | TypeScript + DOM | C++ + CDP |
| 协议层 | CDP 直连 | Playwright API | DOM API / Chrome Extension | CDP / WebDriver |
| 工具层 | CLI + Skill | Python 框架 | CDN / npm 包 | 多语言 SDK |
| Agent层 | Claude Code Skill | LangChain / 自定义 | 内置 Agent + MCP | MCP Server |

### Token 效率对比

> ⚠️ agent-browser 数据来自第三方测试，置信度 INFERRED

| 操作场景 | agent-browser | browser-use | page-agent | Playwright MCP |
|----------|---------------|-------------|------------|----------------|
| 页面快照 | 500-800 | 5K-15K | 低（页面内） | 8K-50K |
| 点击操作 | 200-300 | 3K-8K | 低 | 8K-12K |
| 表单填写 | 300-500 | 4K-10K | 低 | 8K-10K |
| 多步骤任务 | 极低 | 中高 | 低 | 极高 |

**结论**：agent-browser 在 Token 效率上有数量级优势，page-agent 因页面内运行也很低，传统 Playwright MCP 消耗最大。

## 生态位关系

这四个工具不是简单的竞争关系，而是分层互补：

```
Playwright (底层引擎)
    ↓
browser-use (自然语言封装)    agent-browser (AI 原生 CLI)
    ↓
page-agent (页面内集成)
```

- **Playwright** 是基础设施，其他工具可能直接或间接依赖它
- **browser-use** 和 **agent-browser** 是同层竞争，都面向 AI Agent 浏览器控制
- **page-agent** 走的是不同路线（页面内 vs 页面外），更偏向产品集成

### 组合使用建议

1. **AI Agent + 浏览器自动化**：agent-browser（主） + Playwright（复杂场景兜底）
2. **SaaS 产品 + AI Copilot**：page-agent（产品内） + agent-browser（外部辅助）
3. **测试自动化 + AI**：Playwright（主力） + browser-use（AI 增强）
4. **研究探索**：browser-use（Python 生态好，快速实验）

## 发展趋势判断

1. **AI 原生设计是趋势**：agent-browser 的快速增长证明了 AI 原生设计的市场需求
2. **Token 效率越来越重要**：随着 Agent 任务复杂化，Token 成本将成为关键选型因素
3. **页面内外融合**：page-agent 的页面内模式和外部 CLI 模式可能走向融合
4. **协议标准化**：MCP 等协议将推动浏览器控制能力的标准化和可移植性

## 尚待解决的问题

1. **多浏览器支持**：agent-browser 和 page-agent 目前主要支持 Chrome
2. **复杂交互支持**：拖拽、右键菜单、Canvas 等复杂操作仍是挑战
3. **性能基准数据**：缺乏统一的 benchmark 对比各工具性能
4. **生产环境稳定性**：新项目（agent-browser）的长期稳定性有待验证
5. **安全边界**：AI 控制浏览器的安全风险和权限管理机制
