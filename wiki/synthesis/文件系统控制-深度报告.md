# 文件系统控制 深度报告

> 综合自 5 篇素材 | 生成日期：2026-06-28

## 背景概述

在 AI Agent 快速发展的今天，如何让 AI 在长时间任务中保持一致的上下文、记忆和状态管理能力，成为工程实践中的核心挑战。传统的对话历史存在线性、嘈杂、包含过时纠错信息等问题；向量数据库虽然能解决部分检索需求，但复杂且检索质量不稳定。

文件系统作为 Unix 哲学的核心概念，在 AI Agent 时代被重新审视。Manus 提出的"Using File System as a Context"理念，以及 Claude Code 用"文件系统 + bash"的简单方案击败复杂 embedding 索引路线的事实，证明了这一方向的可行性。

上下文工程（Context Engineering）作为新兴的工程学科，正在代替提示词工程成为 AI Agent 开发的主流方法论。多数 AI Agent 的失败，并非模型能力的失败，而是上下文工程的失败。

## 核心观点

### 1. 文件系统是 Agent 长期记忆和复杂推理的最佳载体

对话流只适合短暂的指令交互，而文件系统才是智能体长期记忆和复杂推理的最佳载体。其核心优势包括：

- **单一真理来源**：文件是经过整理的、当前的「最新状态」，AI 每次行动前应主要读取文件，而非回溯聊天记录
- **可恢复性**：重启后读取文件即可恢复状态，即使关闭终端、重启电脑，AI 再次启动时读取文件，立刻知道进度条在哪里
- **原子性**：操作具有原子性，易于版本控制
- **结构化**：目录结构提供天然的层次化组织方式

### 2. 上下文工程的六大核心原则

| 原则 | 核心思想 | 关键实践 |
|------|---------|---------|
| 文件即单一真理来源 | 对话历史嘈杂，文件才是最新状态 | AI 每次行动前读取文件而非回溯聊天记录 |
| 状态显式化 | 通过外部化记忆记录进度 | ToDo 文件（task_plan.md）标记 [x] / [ ] 进度 |
| 上下文窗口极简主义 | 只喂当前步骤必要信息 | 信息拆分到 notes.md、plan.md、output.md，按需加载 |
| 思考与行动分离 | 先想再做，避免代码库污染 | 先在 Notes 写调研结果，确认后再改正式代码 |
| 围绕 KV-Cache 设计 | 所有 Prompt 为缓存命中率让路 | 保持前缀稳定、只追加不修改、确定性序列化 |
| 掩码而非移除工具 | 保留所有工具定义 | 在解码阶段通过修改 Logits 屏蔽非法工具 |

### 3. Agent 设计是关键，而非模型本身

Claude Code 的成功在于极致克制和精心设计的 Agent 架构，而非单纯依赖更大上下文窗口。即使把 Cursor 或 GitHub Copilot 的底层模型换成 Claude 4，整体体验往往还是跟 Claude Code 差一截。

Claude Code 采用「分而治之」和「按需加载」的策略，像组建专业团队，让每个成员专注自己的任务。

### 4. 极致简单化哲学

Claude Code 像一个真正的结对程序员——够自主、能完成该做的事情，但不至于过分炫技让人产生失控感。工具设计三原则：**简单直接、可预测、可控**。

### 5. 文件系统与向量数据库的分工

两者不是替代关系：
- **文件系统**：结构化、长期记忆、状态管理
- **向量数据库**：非结构化内容检索、RAG

## 不同视角对比

### Manus vs Claude Code

| 维度 | Manus | Claude Code |
|------|-------|------------|
| 核心定位 | 通用 AI Agent | AI 编程工具 |
| 文件系统实践 | ToDo 文件、外部化记忆 | 文件系统 + bash 方案 |
| 上下文原则 | 六大原则完整实践 | 分而治之 + 按需加载 |
| 框架演进 | 已重建 Agent 框架四次 | 极致简单，专注核心 |

### 提示词工程 vs 上下文工程

| 维度 | 提示词工程 | 上下文工程 |
|------|-----------|------------|
| 关注点 | 单次 Prompt | 整个对话上下文 |
| 粒度 | 粗粒度 | 细粒度 |
| 方法 | 优化 Prompt 本身 | 管理信息来源和流动 |
| 失败表现 | Prompt 不够清晰 | 上下文丢失、注意力分散 |

### 状态管理方式对比

| 方式 | 状态保存 | 重启恢复 | 可靠性 |
|------|---------|---------|--------|
| 对话历史 | 自动 | 依赖上下文窗口 | 低（可能丢失） |
| 向量数据库 | 自动 | 需要检索 | 中（检索质量不稳定） |
| 外部化文件 | 显式写入 | 直接读取文件 | 高（完全可控） |

## 技术栈全景

### 原子能力层
- **读取（Read）**：读取文件内容作为上下文
- **写入（Write）**：将状态、进度写入文件
- **列表（List）**：列举目录结构，组织信息
- **搜索（Search）**：快速定位相关文件

### 核心文件结构
```
task_plan.md    → 记录目标、步骤、进度、下一步
notes.md        → 调研资料、中间代码、临时想法
[deliverable].md → 最终纯净输出，不含思考过程
```

### 性能优化层（KV-Cache）
- **保持前缀稳定**：不在 System Prompt 中放入动态内容
- **只追加不修改**：历史交互记录一旦生成不修剪
- **确定性序列化**：JSON 对象 Key 排序必须固定
- **掩码而非移除工具**：保留所有工具定义，通过 Logits 屏蔽

## 知识脉络

### 演进路径

```
提示词工程（Prompt Engineering）
    ↓ 局限：单次交互，无法管理长期上下文
上下文工程（Context Engineering）
    ↓ 核心：以文件为载体实现状态管理
        文件系统即 Meta Tool
            ↓ Unix 理念复兴
        planning-with-files 项目（23K+ stars）
            ↓ 实践验证
        Manus 六大原则 + Claude Code 极致克制
```

### 关键里程碑

1. **Unix 理念复兴**：一切皆文件，open/read/write/close 统一接口
2. **上下文工程诞生**：Manus 提出并实践六大原则
3. **planning-with-files 项目**：5 个月 23K+ stars，验证文件系统方案可行性
4. **Claude Code 成功**：用简单方案击败复杂 embedding 索引路线
5. **行业共识形成**：上下文工程正在代替提示词工程成为主流

## 尚待解决的问题

1. **跨会话上下文连贯性**：如何确保 Agent 在多会话切换时无缝衔接，仍有待最佳实践
2. **文件系统操作的原子性保障**：在复杂任务中，如何确保文件系统操作的原子性和一致性
3. **性能与精度的平衡**：KV-Cache 优化策略在复杂上下文场景下的边界条件
4. **工具定义的动态掩码**：解码阶段 Logits 修改的实现方案尚无开源标准
5. **与向量数据库的深度集成**：文件系统与向量数据库的协同机制仍有探索空间
