# 上下文压缩

> 减少 AI Agent 上下文窗口占用，提升 Token 利用效率

## 定义

上下文压缩是在内容到达 LLM 之前或之后，通过算法减少 Token 数量的技术。核心目标：解决 AI Agent 上下文窗口污染问题。

## 两类问题

### 1. 窗口大小问题

行业主流方向：
```
更大的上下文窗口 —— 128K → 200K → 1M → 2M token
      │
      每个版本都在堆：更长的 KV 缓存、更大的注意力计算
      │
      成本线性增长
```

<!-- confidence: EXTRACTED -->
<!-- evidence: Headroom 技术分析 -->

### 2. 内容质量问题

Headroom 的方向：
```
压缩输入 —— 60-95% token 节省
      │
      不改变模型、不改变 API
      │
      在输入到达之前减量
```

## 压缩类型

### 可逆压缩

- 原始内容缓存 + 检索恢复
- LLM 可按需检索原文
- 适用场景："偶尔需要完整数据"（Agent 工作流常态）

代表项目：Headroom（CCR）、Context-mode（FTS5）

<!-- confidence: EXTRACTED -->
<!-- evidence: Headroom 技术分析 -->

### 不可逆压缩

- 更快、更简单
- 原文丢失——无法恢复
- 适用场景："看一眼就行"

代表项目：RTK、lean-ctx、LLMLingua

## 隐形成本

```
一个典型的 SRE 调试会话：
Agent 读取日志文件（17,892 token）
Agent 执行 grep 命令（4,321 token）
Agent 检查配置文件（8,456 token）
Agent 查看指标面板（6,234 token）
Agent 汇总分析（28,791 token）
                           ─────────
总计                      65,694 token
```
