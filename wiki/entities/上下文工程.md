# 上下文工程

> AI Agent 开发中对上下文进行精心设计和管理的工程实践

<!-- confidence: EXTRACTED -->
上下文工程（Context Engineering）是指在 AI Agent 开发中，对上下文进行精心设计和管理的工程实践。与提示词工程不同，上下文工程关注整个对话上下文的管理，而非单次 Prompt 的优化。

## 核心原则

### 1. 文件即单一真理来源
<!-- confidence: EXTRACTED -->
对话历史是线性的、嘈杂的，且包含大量过时的纠错信息。而文件是经过整理的、当前的「最新状态」。AI 每次行动前，应该主要读取文件，而不是回溯聊天记录。

### 2. 状态显式化
<!-- confidence: EXTRACTED -->
通过外部化记忆（ToDo 文件）记录进度，即使重启电脑也能恢复。

### 3. 上下文窗口极简主义
<!-- confidence: EXTRACTED -->
只喂给 AI 当前步骤必要的信息，保持注意力集中。信息拆分到不同文件，按需加载。

### 4. 思考与行动分离
<!-- confidence: EXTRACTED -->
先想再做，避免写到一半发现思路错了导致代码库被污染。

### 5. 围绕 KV-Cache 进行设计
<!-- confidence: EXTRACTED -->
所有 Prompt 设计都要为缓存命中率让路：
- 保持前缀稳定，不在 System Prompt 中放入动态内容
- 只追加不修改：历史交互记录一旦生成不修剪
- 确定性序列化：JSON 对象的 Key 排序必须固定

### 6. 掩码而非移除工具
<!-- confidence: EXTRACTED -->
保留所有工具定义，在解码阶段（Decoding）通过修改 Logits 屏蔽当前不合法的工具。

## 与提示词工程的区别

| 维度 | 提示词工程 | 上下文工程 |
|------|-----------|------------|
| 关注点 | 单次 Prompt | 整个对话上下文 |
| 粒度 | 粗粒度 | 细粒度 |
| 方法 | 优化 Prompt 本身 | 管理信息来源和流动 |

## 为什么重要

**多数 AI Agent 的失败，并非模型能力的失败，而是上下文工程的失败。**

上下文长度影响 AI 注意力分散，信息质量比信息数量更重要。
