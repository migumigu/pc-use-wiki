## 核心问题：Agent 的"金鱼记忆"

每次新 Session 都要重新介绍项目背景："我们用 jose 做 JWT 中间件"、"测试跑在 pytest with xdist"...

Agent 能写代码、修 bug、重构，**但它记不住任何东西**。

<!-- confidence: EXTRACTED -->

## 四层记忆架构

### 1. Working Memory（工作记忆）
- Agent 当前 session 每次 tool call 的原始记录
- PostToolUse hook 触发 → SHA-256 去重 → 隐私过滤 → 存储原始观测

### 2. Episodic Memory（情景记忆）
- Session 结束后自动压缩
- 用 LLM 从原始观测里抽取结构化事实

### 3. Semantic Memory（语义记忆）
- 跨 session 的事实沉淀
- 置信度随反复验证上升

### 4. Procedural Memory（程序记忆）
- 被反复验证的流程和决策模式
- 相当于 Agent 的肌肉记忆

## 记忆生命周期管理

- **置信度评分**：随时间衰减（艾宾浩斯遗忘曲线）
- **访问频率强化**：频繁访问的记忆被强化
- **长期不被访问降权**
- **冲突检测**：新旧信息冲突自动标记

## 三层检索融合

| 搜索方式 | 能做到的 | 盲区 |
|----------|----------|------|
| BM25 | 精确找到关键词 | 找不到同义词 |
| 向量搜索 | 语义相近 | 项目特定命名懵了 |
| 知识图谱 | 关联遍历 | 需正确提取实体 |

结果用 **RRF (k=60)** 融合，R@5 从 86.2% 拉到 **95.2%**。

## 设计哲学

> **把 memory 从一个「存储问题」变成了一个「检索 + 生命周期管理问题」。**

Agent 不缺存储。它需要的是「在正确的时机，用正确的方式，把正确的东西拿回来」。

## 适用人群

### 适合
- 每天在 Claude Code/Cursor 里泡 4+ 小时
- 同时用多个 Agent（跨 Agent 共享记忆）
- 需跨会话决策追踪

### 不适合
- 短平快一次性任务
- 简单项目（200 行 CLAUDE.md 更简单）

## 设计克制

默认关闭：
- LLM 自动压缩（太烧 token）
- slot 反射
- 上下文自动注入

**设计者在做一个「你可以信任的工具」，而不是「默认把你吃干抹净的平台」。**
